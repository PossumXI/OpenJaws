# Security Audit Note: XAA IdP Login Token Handling

## Issue Identified
- The `openjaws mcp xaa login` command accepts a pre‑obtained JWT via the `--id-token <jwt>` option.
- The JWT is passed as a command‑line argument which can be leaked through:
  - Shell history (`.bash_history`)
  - Process listings (`ps aux`), especially on multi‑user systems.
  - Debug logs or crash reports that may capture argv.

## Recommended Mitigation
- Support reading the token from **stdin** when the argument value is `-` (e.g., `echo $TOKEN | openjaws mcp xaa login --id-token -`).
- This approach avoids storing the secret in the command line and aligns with the existing TODO comment on lines 162‑164.
- Ensure the implementation reads the full stdin stream before proceeding and validates that the token is non‑empty.

## Additional Hardening Suggestions
1. **Validate JWT format** before caching to prevent malformed data.
2. **Set restrictive file permissions** on any cached token storage.
3. **Add explicit warnings** in the CLI help output about the security implications of using `--id-token` with a literal value.

## Next Steps
- Implement the stdin handling as described.
- Update the help text to mention the `-` shortcut.
- Add unit tests covering both inline and stdin token inputs.
