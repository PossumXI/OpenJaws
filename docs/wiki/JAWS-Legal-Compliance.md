## Billing And Refunds

The intended JAWS IDE subscription policy is final sale for paid subscription periods and delivered services, except where a refund, cancellation right, chargeback right, or other remedy is required by applicable law or payment-network rules.

This wording is intentional. The FTC describes specific cancellation and refund obligations for certain sale types, including the Cooling-Off Rule for certain off-premises sales and the Mail, Internet, or Telephone Order Merchandise Rule for delayed or cancelled merchandise orders. Keep the public policy clear, conspicuous, and reviewed by counsel before launch.

- FTC Cooling-Off Rule: `https://www.ftc.gov/legal-library/browse/rules/cooling-period-sales-made-home-or-other-locations`

## Security Boundary

- Signed updater artifacts are required for public releases.
- Updater endpoints must use HTTPS on both `qline.site` and `iorch.net`.
- The private updater signing key must stay in CI secrets only.
- Desktop release config scans the JAWS app surface for legacy provider branding and fails closed.
- Public copy checks must pass before public deploys so local paths, raw receipts, and token-shaped content do not leak.

## Developer Verification

Run these before public release promotion:

```powershell
bun run jaws:verify
bun run jaws:release:check
bun run jaws:mirror:check --json
bun run jaws:soak -- --duration-ms 300000 --users 5000
bun run showcase:copy:check
```

The timed soak verifies logo presence, docs/legal availability, release metadata, updater security, Agent Watch runtime bridge wiring, Slow Guy mechanics, Hold'em showdown mechanics, and a synthetic user-presence scaling loop.
